# CLS 认知循环系统

> **将 LLM Agent 推理结构化为可验证的 6 步闭环 —— 配备双 AI 验证闸门、熔断板安全层、跨会话持续学习。**
> 
> 作者：**认知工坊**  
> 联系方式：QQ 1419456542 · 邮箱 zyc2018@mail.ustc.edu.cn

CLS（Cognitive Loop System，认知循环系统）是一个 harness 层推理框架，将大语言模型的不确定性输出转化为可验证、自改进的管线。不同于让模型单次生成后即结束，CLS 将每个任务包裹在 6 步认知循环中，强制执行态势感知、结构化执行、对抗学习、抽象泛化、上下文持久化和轨迹记录。该架构于 **2026 年 5 月 25 日** 首次落地代码 —— 比 2026 年 6 月 "Loop Engineering" 概念进入公共 AI 工程讨论约早 2 周。

## 起源故事

2026 年 5 月，在开发 Claude Code harness 系统的过程中，一个模式反复出现：单次 LLM 交互不断撞上相同的墙 —— 上下文漂移、自指幻觉、缺乏对刚完成操作的持久记忆。跨多个会话的任务会退化，因为没有结构性的"脊柱"来承载状态。

**2026 年 5 月 25 日**，commit `73ee3a84` 将 6 步认知循环首次落地为代码。六步分别为：

1. **态势感知** —— 读取活跃上下文，将用户意图与认知轨迹匹配
2. **任务执行** —— 规模判断、起飞前检查、执行、收尾
3. **关联学习** —— 将新知识与已有模式匹配
4. **抽象泛化** —— 从具体经验中提取可迁移的模式
5. **上下文持久化** —— 将检查点写入磁盘，实现跨会话连续性
6. **轨迹更新** —— 记录 Δq（问题演化）和 Δp（进展变化）

两周后，2026 年 6 月初，Anthropic 的 Boris Cherny（Claude Code 之父）和 Google 的 Addy Osmani 开始公开讨论 "Loop Engineering" —— AI harness 应自动组装上下文、生成 prompt、执行、观察、循环，取代手写 prompt 的旧范式。这一概念引起共鸣，因为它验证了 CLS 已经走上的方向。

**时间线：**

| 日期 | 事件 |
|------|------|
| 2026-05-25 | CLS 6 步认知循环首次写入代码（commit `73ee3a84`） |
| 2026-06-02 | 熔断板（6 个独立熔断器）激活 |
| 2026-06-03 | 双 AI 闸门（生成/验证分离）强制执行 |
| 2026-06-07 | Addy Osmani 发布《Loop Engineering》博文，正式命名该范式 |
| 2026-06 中旬 | Boris Cherny 与 Peter Steinberger 公开演讲推动 loop-based harness 设计 |

**CLS 在基础 Loop 概念之上的增量：**

- **生成/评估分离** —— 不只是"不评估自己的输出"，而是引入*第二个独立模型*（Qwen）作为验证者，统计学保证：P(DS 错) × P(QW 错) = 双重幻觉率（如 0.1 × 0.1 = 1%）
- **熔断板层** —— 6 个独立熔断器在认知系统*之外*运行，纯 stdlib 编写，不 import 任何受保护的推理模块
- **符号压缩** —— 5 万 token 的对话压缩为 3 个数字 + 1 个状态词，通过拓扑熵、Perron-Frobenius 谱和禁用词状态机实现
- **跨窗口协作** —— 多个 CC 会话共享状态文件，互相感知焦点，避免域冲突

## 核心概念

### 6 步认知循环

每个任务 —— 无论是一个简单问答还是数小时的工程设计 —— 都强制通过六步。这不是可选的；harness 在结构层面强制执行。

| 步骤 | 名称 | 做什么 |
|------|------|--------|
| ① | **态势感知** | 读取 `active_context`、`cog_thread` 和 `trajectory.json`。将用户关键词与最近 5 条轨迹匹配。通过跨窗口钩子感知其他 CC 窗口状态。 |
| ② | **任务执行** | 规模判断（简单/中等/复杂）→ 起飞前检查（熔断板、前提验证）→ 带监控的执行 → 收尾。 |
| ③ | **关联学习** | 新知识与已有模式匹配。关联相似的历史任务。标记矛盾。 |
| ④ | **抽象泛化** | 具体经验蒸馏为可迁移模式。"修 bug X 改了 Y" 变为 "Z 类 bug 需要 W 模式"。 |
| ⑤ | **上下文持久化** | 活跃上下文、当前焦点和检查点数据写入磁盘。这是跨会话连续性的基础 —— 系统醒来时知道自己之前在做什么。 |
| ⑥ | **轨迹更新** | 记录 Δq（问题如何演化）和 Δp（进展如何变化）。轨迹文件作为可搜索的认知运动历史持续增长。 |

循环闭合：步骤 ⑥ 直接反馈到下一个任务的步骤 ①。这不是线性流水线 —— 它是一个每次迭代都更紧密的循环。

### 双 AI 闸门

核心统计学洞察：两个独立模型同时犯同一错误的概率是一个模型单独犯错的乘积。

- **生成者（DeepSeek）：** 创建设计、编写代码、捕获知识
- **验证者（Qwen）：** 用全新上下文独立验证这些产出

闸门以三种模式运行：
1. **设计检查** —— 验证 CAD 设计完整性和几何合理性
2. **知识检查** —— 验证知识声明的一致性和可复现性
3. **数值检查** —— 独立重新计算并检查数量级

如果 Qwen 不可用，系统降级到 Anthropic 的 Haiku 模型作为备用审计者。如果两者都不可用，闸门进入 `unavailable` 状态默认放行 —— 但记录降级事件。

### 熔断板

熔断板是一个在认知系统*之外*运行的安全层。它不 import 任何推理模块。纯 Python stdlib。它不能被它保护的系统修改。

| 熔断器 | 防护目标 | 动作 |
|--------|---------|------|
| `WRITE_PROTECT` | 自指修改核心文件 | 阻止写入受保护路径 |
| `RECURSION_LIMIT` | 无上限自指嵌套（最大深度 5） | 截断 + 30 分钟冷却 |
| `TOKEN_BUDGET` | API 费用失控（日 200 万/会话 50 万） | 阻止 API 调用 |
| `PARALLEL_CAP` | 同时推进过多破坏性变更（上限 3） | 阻止新并行操作 |
| `CHECKPOINT_REQUIRED` | 重大变更前无回滚点 | 每 300 秒强制保存 |
| `PROXY_PURITY` | 代理层语义变换 | 只允许字段删除 |

熔断板设计了硬件迁移路径：`FuseBackend` 抽象接口意味着软件后端可以替换为硬件实现，而不改变任何调用代码。

### 符号动力学审计

每次工具调用都经过实时审计守护进程检查：
- **禁用词模式** —— 捕获已知失败特征的正则状态机
- **域特定转移矩阵** —— 编码合法 token 序列的加权转移图
- **拓扑熵** —— 单个数字衡量输出空间的"混沌"程度（0 = 完全确定，ln(N) = 完全随机）

守护进程在 **200 毫秒内** 处理每次检查，返回裁决：`allow`（通过）、`ask`（标记人工审核）或 `deny`（阻止）。一个 5 万 token 的对话最终压缩为 3 个数字（拓扑熵、谱半径、告警计数）+ 1 个状态词，供人类监督。

### 跨窗口感知

当多个 CC 会话同时运行时，它们共享一个状态文件（`state/cross_window_context.json`）。每个窗口：
- 在步骤 ①（态势感知）中**窥视**其他窗口的焦点
- 在步骤 ②（任务执行开始）中**宣告**自己的焦点
- 在承诺任务前**检查域冲突** —— 如果另一个窗口已在同一域工作，协调而非冲突

这防止了经典的多 Agent 问题：两个会话不知道对方存在，同时编辑同一文件或追求矛盾的目标。

### 事实锚定协议

系统做出的每一个声明必须锚定到具体的文件路径和字段值。没有文件引用的声明视为无根据。这是结构层面的抗幻觉机制：

- **声明：** "系统健康。"
- **驳回。** 无文件路径，无字段，无证据。
- **声明：** "`state/session_health.json#status.msgs_current` = 42，低于 compact 阈值 150。"
- **接受。** 可通过读取引用文件验证。

## 快速开始

### 前置条件

CLS 是一个 **harness 层架构** —— 不是独立应用，而是一组协议、脚本和配置模板，与 LLM Agent 宿主（当前为 Claude Code）集成。要实验或适配 CLS，你需要：

- 一个可工作的 Claude Code 安装（架构设计为宿主无关；见 `docs/host-independence.md`）
- Python 3.10+ 用于安全脚本和符号动力学引擎
- （可选）第二个 LLM API 端点用于双 AI 闸门（架构支持 Qwen 或任何独立模型）

### 集成点

CLS 在 LLM Agent harness 的三个层面集成：

1. **配置层** —— 编码 6 步循环为行为指令的 `CLAUDE.md` / 系统 prompt 模板
2. **钩子层** —— 在每次工具调用前运行熔断板检查和符号审计的 PreToolUse/PostToolUse 钩子
3. **守护进程层** —— 长期运行的后台进程（符号动力学守护进程、激活监听器、跨窗口状态管理器）

参见 `CLAUDE_templates/` 获取示例 harness 配置，`docs/integration-guide.md` 获取分步说明。

### 快速验证

```bash
# 运行熔断板自检（纯 stdlib，无依赖）
python scripts/safety/fuse_board.py --test

# 在样本数据上运行符号动力学引擎
python scripts/safety/symbolic_dynamics_engine.py --test

# 检查双 AI 闸门能否初始化（需要 API 访问）
python scripts/safety/qwen_gate.py --health
```

### 设计哲学

CLS 基于 11 条设计原则（见 `rules/philosophy/design-philosophy.md`）。最重要的三条：

1. **接口 > 实现** —— 签名固定，后端可换。熔断板抽象接口意味着今天的软件实现可以变成明天的硬件实现，不改变一行调用代码。
2. **三元乘法，不是 AGI** —— CLS 不为创造通用人工智能。它为 `AI × CLS × 人类` 形成稳定闭环。AI = 算力与生成（引擎），CLS = 记忆与约束（底盘），人类 = 目标与价值判断（方向）。任一为零则系统归零。
3. **稀疏激活，共享兜底** —— 安全层永远在线。域特定技能按需加载。路由失败时基线仍在。

## 贡献指南

CLS 是一个通过使用不断改进的活架构。欢迎多种形式的贡献：

- **Bug 报告：** 开 issue 描述失败模式、预期行为和最小复现案例。
- **架构提案：** 重大变更请先开讨论 issue。CLS 对关注点分离有很强的立场（生成 vs 验证、安全 vs 认知、接口 vs 实现）。尊重这些边界的提案更容易被接受。
- **文档：** 对文档、示例和集成指南的改进始终有价值。
- **新熔断器类型：** 如果你遇到了可以用新熔断器预防的失败模式，请提出：（a）它防止的事故，（b）阈值逻辑，（c）触发时的动作。

### 提交前

1. 阅读 `rules/philosophy/design-philosophy.md` 理解架构约束。
2. 确保你的改动不会将认知模块导入安全层（熔断板 stdlib-only 规则）。
3. 如果添加新熔断器，提供驱动它的历史事故 —— CLS 中的每个熔断器都对应一次真实生产事故。
